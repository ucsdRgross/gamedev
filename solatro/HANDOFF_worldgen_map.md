# Handoff: worldgen world-map integration + run/persistence layer

Last updated 2026-07-07. This document lets a new contributor pick up the world-map / run /
persistence work. Companion docs: [ARCHITECTURE_REVIEW.md](ARCHITECTURE_REVIEW.md) §1.5
(architecture), [DESIGN_DOC.md](DESIGN_DOC.md) §2 & §15 (design intent). The full planning
history lives in the plan file `~/.claude/plans/goal-is-to-embed-cached-biscuit.md`.

## 1. What this feature is

Replace the old placeholder "triangle map" with a **traversable procedural world map**
(Slay-the-Spire style) generated by the **worldgen addon**, plus the run/progression layer
around it: node roles (shows vs booster packs), a 3-act game win/loss, a fame/luck system,
endless reverse laps, and full to-disk persistence of the run (including mid-game undo
history). All of that is **implemented**; remaining work is polish/tuning/follow-ups (§6).

## 2. Status — done

- **Addon embedded**: `addons/worldgen/` (vendored snapshot; canonical home is the separate
  `worldgen` project — re-copy to update, never edit the copy). Plugin enabled in
  `project.godot`; `RunManager` autoload registered.
- **Map screen** (`Levels/map.gd` + `map.tscn`): plain 2D `WorldMap2D` + `Camera2D`
  (drag-pan, wheel-zoom, follows token) + player token, driven by `WorldMapController`.
  Old 3D-tilted SubViewport presentation removed.
- **Traversal**: direction-aware reachability (reverse adjacency on odd laps), 4 edge
  visual states (traveled=gold / next=highlighted / usable=normal / unusable=hidden),
  **mouse AND keyboard/controller** node selection, camera follow.
- **Node roles** (`Scripts/Map/map_node_roles.gd`): deterministic from seed — start/end are
  anchors (end = boss), ~1 rank per 3 becomes boosters, rest are shows; written into
  `WorldGraphNode.meta`, re-derived every populate (never persisted).
- **Game loop** (`Levels/game.gd`): a show = exactly 3 submits ("acts"). After the 3rd,
  win if `total_score >= goal` → fame → back to map; loss → run over → menu. Each submit
  pays `row_total × col_total` (`GameData.apply_act_score`) and clears the lower board
  (`discard_lower_board`; Entrance/upper zone kept). Win/loss use an explicit Continue
  button (mouse + keyboard).
- **Boosters** (`Cards/Types/booster_template.gd`, `type_booster_basic.gd`): entering a
  booster node opens a take-all `ChoiceViewer` (all 5 cards added). Every card gets a base
  type; luck (from fame) upgrades stamp/skill/type odds. Hover panel previews possible
  contents with a per-card modifier inspector.
- **Fame / luck / goals** (`Scripts/run_manager.gd`): win adds full `total_score` to fame
  (overscore tax removed 2026-07, SCORING_MATH_PLAN §8c′); goals come from the §15b
  N̂-curve; luck (saturating curve of fame) biases booster generation. All tunables live
  in `Scripts/player_settings.gd` "Balance" groups.
- **Endless laps**: end node = boss; win → summary popup → direction reverses on the same
  graph, goals rescale per lap, traveled history accumulates (gold).
- **Menu** (`Levels/menu.gd`): Play unfolds New Run / Continue; New Run → `DeckPicker`
  (lists `deck1..deck10`); Continue gated on `RunManager.has_save()`.
- **Persistence** (`user://run_save/`): `run.tres` (RunState incl. full undo history) +
  `map/` (worldgen bake). **Every action saves** (anti-cheat: closing can't rewind);
  writes are coalesced on a **background thread**; quitting mid-show resumes the exact
  board + undo stack. Loss clears the save.
- **Tests** (`Tests/`, run `res://Tests/all_tests.tscn`): `Engine/test_act_score`,
  `Map/test_run_manager`, `Map/test_map_roles`, `Map/test_map_traversal`,
  `UI/test_ui_viewers`. All follow the non-freezing `check()` pattern.
- **Cleanup**: deleted `triangle_map.gd/.tscn`, `gui_3d.gd`; `player_save.gd` trimmed to
  the Deck Maker dev tool only. Docs updated (ARCHITECTURE_REVIEW N10/N11/N12, DESIGN_DOC

## 4. Key mechanics to understand before editing

- **Node roles are never saved.** `graph.json` doesn't round-trip `WorldGraphNode.meta`, so
  `MapNodeRoles.assign(overlay, world_seed, run)` re-derives them (deterministically from
  the seed) on every `graph_populated` and every lap change. Don't try to persist them.
- **Lap direction.** Even lap = forward (start→end), odd lap = reversed (end→start). The
  controller builds a reverse-adjacency map once per populate; `traveled` history is always
  stored in forward-edge orientation `(src, dst, lap)`.
- **Serialization has two gotchas** (both handled — don't reintroduce):
  1. `CardModifier.data` is a self-cycle (card→modifier→card). ResourceSaver can't write
     cycles. `GameData.to_saveable()` unlinks it; `restore_runtime()` relinks (backref
     always = owning card). `ZoneAdder.card_data` is a forward ref (no cycle) — leave it.
  2. `BigNumber` is RefCounted (not serializable). `GameData` keeps runtime `Array[BigNumber]`
     un-exported and persists each score array as two PARALLEL typed packed arrays —
     `packed_*_mant` (`PackedFloat64Array`) + `packed_*_exp` (`PackedInt64Array`), synced by
     `pack_scores()`/`unpack_scores()`. (Replaced the old `Dictionary` of `[m,e]` pairs:
     typed packed arrays are contiguous and skip the per-pair Variant/Array allocation.)
     Note packed arrays are copy-on-write value types — `pack_scores` assigns built arrays
     back to the fields rather than mutating a parameter.
- **Save cadence.** In-game every action → `RunManager.request_save()` (async, coalesced,
  background thread, temp-file+rename). Map/booster/run-start → `save_run()` (sync). App
  quit → `RunManager._exit_tree()` flushes + joins the thread. The run deck is cached and
  only re-copied on `mark_deck_dirty()` so in-show saves stay cheap.
  - **Temp-file extension gotcha (fixed).** `_write_payload` writes to a temp then renames.
    The temp MUST end in `.tres` (`run.tmp.tres`) — `ResourceSaver.save` deduces its format
    from the extension, so the earlier `run.tres.tmp` failed with `ERR_FILE_UNRECOGNIZED`
    and silently wrote nothing. Result: no `run.tres`, `has_save()` false, Continue disabled.
- **`has_save()` gates on `run.tres` ALONE** (not the map bake). The `map/` bake is a
  deterministic cache of `world_seed` — `WorldMapController.start_run` rebakes it when
  missing — so requiring it made Continue fragile (a deleted/half-written cache disabled an
  otherwise-recoverable run). The bake regenerates transparently on the next resume.
- **Pending-action replay (anti-cheat for interrupted actions).** Board-mutating buttons that
  resolve asynchronously (Submit → `on_run_scorer`, Next → `on_next`) persist a
  `RunState.pending_action` marker with the pre-action board *before* they await. If the app
  is killed mid-resolution, `Game._resume_show` replays that action from the restored
  pre-action board (`_replay_pending_action`) with input locked — the player can't dodge a
  Submit/Next by quitting. `save_state` clears the marker the instant an action commits.
  Requires these actions stay deterministic (scoring has no RNG; draws come from the ordered
  `draw_deck`). A fully-submitted show (`submits_used >= MAX_SUBMITS`) re-resolves on resume.
  UI-only buttons (deck viewer, Continue screen) carry no board state and need no marker.
- **Save size / cadence.** Each committed action appends one saveable `GameData` snapshot to
  `game_history`; a realistic 72-card board is ~34 KB as text `.tres`, so `run.tres` grows
  ~34 KB per action (every snapshot re-stores all cards). A long show reaches a few hundred
  KB–low MB. If this becomes a concern, cap undo depth or move to an append-only per-snapshot
  journal (see follow-ups) — the format is text/verbose by design (readable, ResourceSaver).
- **Per-act score gutters reset in `apply_act_score`.** The `scores_row_*/scores_col`
  BigNumber gutters accumulate during an act (`plus_equals`); `apply_act_score` clears all
  three alongside `row_total/col_total`, or the next act stacks onto them. Their LABELS only
  resync via `PlayArea.update_score_controls()` (the revision-bump rebuild path does NOT touch
  them) — called from `Game._perform_submit` and on resume.
- **Card viewers = a container + a `CardsViewer`.** DeckViewer / ChoiceViewer / MapHoverPanel
  each contain a `CardsViewer` (`UI/cards_viewer.gd`) that owns the card list (`populate`/
  `clear`, optional hover+focus inspector). They can't share a base class (roots differ:
  CanvasLayer / Control / PanelContainer); `ControlCard` is one card. All populate
  synchronously — no fly-in comes from `CardVisual` positioning per `current_context`
  (PLAY_AREA eases; viewers snap/track), not per-viewer deferral.
- **Board rebuild timing.** `CardVisual`s add via `call_deferred` (lets the container lay out
  first → no rebuild fly-in), so a freshly built board isn't animatable for a frame. Resume
  waits on `PlayArea.board_visuals_ready` (check-then-await, not polling) before it replays an
  interrupted action; `anim_jump/anim_reset` also guard a null `offset`.
- **Multi-modal input is a hard project rule.** Every UI must work with mouse + keyboard +
  controller: modals steal focus and restore it on close, `ui_cancel` closes, selectable
  elements are focus stops. See `world_map_controller` `_kb_*`, `DeckViewer`, `ControlCard`.

## 5. How to verify

1. Reopen the Godot editor (the addon + several scenes changed on disk; let it re-import).
2. **Unit tests**: run `res://Tests/all_tests.tscn` — expect zero `[FAIL]` lines.
3. **Playtest checklist** (headless runs are unreliable here — run from the editor):
   - New Run → pick a deck → world map renders (nodes + edges visible, layered над the
     map image); pan/zoom; keyboard arrows cycle reachable nodes; token walks curves;
     abandoned edges vanish, traveled edge turns gold.
   - Game node → 3 acts → win adds fame (HUD) and returns; lose → menu, Continue disabled.
   - Booster node → hover shows possible contents + inspect cards; enter → Take all grows
     the deck (map Deck button).
   - Quit mid-show (close window), relaunch → Continue lands on the **exact** board with
     undo history intact. Make a move + undo, then hard-kill → resume reflects the undo
     (closing can't rewind).
   - Reach the end node → boss (2× goal) → win → summary → reverse lap, bigger goals.

## 6. Open follow-ups / known limitations

- **Entrance drop-down not implemented.** `discard_lower_board` clears the lower zone but the
  upper (Entrance) cards don't drop into the lower zone between acts (DESIGN_DOC §2). Decide
  and implement the intended between-act behavior.
- **Background-save robustness unverified at scale.** `ResourceSaver.save` of a large
  `Array[GameData]` history on a worker thread + thread lifecycle on quit are sound by
  design (independent payload, immutable entries, atomic rename) but not stress-tested.
  Watch the console for threading/serialization errors; consider capping undo depth if
  `run.tres` grows large, or moving to an append-only per-snapshot journal.
- **Win/lose screen font** (226px) clips long "Fame +N" text — inspector tweak.
- **First-card fly-in** in the pack preview: deferred-populate should fix it; confirm on a
  real run, and if it persists, snap `CardVisual` to its anchor on first layout.
- **game.tscn initial focus**: no focus is grabbed when a show starts, so keyboard/controller
  players must click first. Grab focus on a sensible control in `Game._ready`.
- **Fame gating / rarity tiers / tips / hype / fog of war** (DESIGN_DOC §15) are not built.
- **Balance**: all formulas live in the const block at the top of `run_manager.gd`; untuned.
- **Luck → generation**: luck only gates whether a booster card gets a non-null
  stamp/skill/type; there's no real rarity system yet.

## 7. Gotchas that cost time before

- The Godot editor rewrites `.tscn`/`project.godot` on disk when open — do those edits with
  the editor closed, then reopen (see memory `godot-editor-disk-sync`).
- Anything fetched from `WorldMap2D` before `add_child` changes child (render) order; the
  controller pins `overlay.z_index = 1` instead of relying on order (nodes were invisible
  under the map image otherwise).
- Never `bake_to_files()` after `reload_from_bake()` — `graph_export` is empty then and it
  corrupts `graph.json`. Bake exactly once, right after the initial generation, then call
  `WorldMap2D.release_generator()` (added to the addon) to drop the GenWorker/loading nodes.
- `String.contains("")` is not reliably true; GDScript lambdas capture locals by value
  (mutate shared reference types in place). Both bit the test suite.
